## 시나리오 개요

대규모 리서치 작업을 여러 전문 에이전트가 분담하여 수행하는 시스템을 설계합니다. 하나의 오케스트레이터가 작업을 분해하고, 전문 에이전트들이 병렬로 조사한 뒤, 결과를 통합하여 최종 리포트를 생성합니다.

## 관련 도메인

- **Agentic Architecture** (27%): 멀티 에이전트 오케스트레이션, hub-and-spoke
- **Context Management** (15%): 에이전트 간 컨텍스트 공유
- **Tool Design & MCP** (18%): 외부 소스 검색 도구 연동
- **Prompt Engineering** (20%): 연구 질문 분해 프롬프트

## 워크스루

### Step 1: 아키텍처 설계 — Hub-and-Spoke

오케스트레이터(Hub)가 연구 질문을 받아 하위 에이전트(Spoke)에게 분배합니다.

```
[Orchestrator]
    ├── [Scholar Agent]     — 학술 논문 검색
    ├── [News Agent]        — 최신 뉴스 수집
    ├── [Code Agent]        — GitHub 레포 분석
    └── [Synthesis Agent]   — 결과 통합 및 리포트 생성
```

### Step 2: 태스크 분해

오케스트레이터는 복잡한 리서치 질문을 하위 작업으로 분해합니다.

| 분해 전략 | 설명 | 적합한 상황 |
|-----------|------|------------|
| 주제별 분할 | 하나의 주제를 세부 주제로 나누기 | 광범위한 조사 |
| 소스별 분할 | 데이터 소스별로 에이전트 배정 | 다양한 소스 탐색 |
| 깊이별 분할 | 표면 조사 → 심층 분석 순서 | 단계적 리서치 |

<Tip title="설계 포인트">
태스크 분해 시 각 하위 작업은 독립적으로 실행 가능해야 합니다. 작업 간 의존성이 있으면 순차 실행으로 전환하세요. 시험에서 "작업 의존성 판단"에 대한 문제가 출제됩니다.
</Tip>

### Step 3: 에이전트 간 통신

각 에이전트의 결과를 구조화된 형식으로 전달합니다.

```json
{
  "agent": "scholar",
  "status": "complete",
  "findings": [
    {
      "source": "논문 제목",
      "relevance": 0.92,
      "summary": "핵심 발견 내용",
      "confidence": "high"
    }
  ],
  "metadata": {
    "sources_searched": 45,
    "time_spent_ms": 12000
  }
}
```

### Step 4: 결과 통합 및 충돌 해결

Synthesis Agent가 모든 결과를 종합할 때 정보 충돌을 처리합니다.

- **일치하는 정보**: 높은 신뢰도로 포함
- **상충하는 정보**: 양측 출처를 명시하고 신뢰도 순으로 정렬
- **불확실한 정보**: "추가 검증 필요"로 표시

<Warning title="함정 문제">
멀티 에이전트 시스템에서 하나의 에이전트가 실패했을 때 전체 시스템을 중단하는 것은 안티패턴입니다. 실패한 에이전트의 결과를 제외하고 나머지로 부분 결과를 생성하는 graceful degradation이 올바른 접근입니다.
</Warning>
