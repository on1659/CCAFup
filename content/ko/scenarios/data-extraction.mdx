## 시나리오 개요

비정형 텍스트(이메일, 문서, 웹 페이지)에서 구조화된 데이터를 추출하는 시스템을 설계합니다. JSON Schema로 출력 형식을 정의하고, few-shot prompting과 검증 루프를 결합하여 추출 정확도를 보장합니다.

## 관련 도메인

- **Prompt Engineering & Structured Output** (20%): JSON Schema, few-shot, 검증
- **Tool Design & MCP** (18%): 데이터 소스 연동
- **Context Management** (15%): 대용량 문서 처리
- **Agentic Architecture** (27%): 추출-검증 에이전트 루프

## 워크스루

### Step 1: 출력 스키마 정의

추출할 데이터의 JSON Schema를 먼저 정의합니다.

```json
{
  "type": "object",
  "properties": {
    "company_name": { "type": "string" },
    "contact_email": { "type": "string", "format": "email" },
    "revenue": { "type": "number" },
    "fiscal_year": { "type": "integer" },
    "industry": {
      "type": "string",
      "enum": ["tech", "finance", "healthcare", "retail", "other"]
    }
  },
  "required": ["company_name", "contact_email"]
}
```

<Tip title="설계 포인트">
JSON Schema의 `enum`을 활용하면 추출 결과를 미리 정의된 값으로 제한할 수 있습니다. 자유 텍스트보다 enum 기반 필드가 후속 처리에 훨씬 안정적입니다. 시험에서 "스키마 설계의 제약 조건"을 묻는 문제가 출제됩니다.
</Tip>

### Step 2: Few-shot 프롬프트 설계

2-3개의 예시를 포함하여 추출 정확도를 높입니다.

| 전략 | 설명 | 장점 |
|------|------|------|
| Zero-shot | 예시 없이 스키마만 제공 | 빠른 구성 |
| Few-shot | 2-3개 입출력 예시 포함 | 정확도 향상 |
| Many-shot | 5개 이상 다양한 예시 | 엣지 케이스 처리 |

### Step 3: 검증 루프

추출된 데이터의 정확성을 검증하는 다중 레이어 검증 시스템입니다.

1. **스키마 검증**: JSON Schema 준수 여부 확인 (프로그래매틱)
2. **형식 검증**: 이메일, 날짜, 통화 등 형식 일치 여부 (정규식)
3. **의미 검증**: 추출된 값이 문맥상 합리적인지 확인 (LLM 재검증)
4. **교차 검증**: 여러 필드 간 일관성 확인 (매출 vs 기업 규모)

```
추출 → 스키마 검증 → 형식 검증 → 통과하면 저장
                              → 실패하면 재추출 (최대 3회)
                              → 3회 실패 시 수동 리뷰 큐
```

### Step 4: 대용량 문서 처리

컨텍스트 윈도우를 초과하는 대용량 문서 처리 전략입니다.

- **청킹**: 문서를 논리적 단위(섹션, 페이지)로 분할
- **맵-리듀스**: 각 청크에서 추출 → 결과 병합
- **우선순위 처리**: 핵심 섹션 먼저 처리, 필요시 추가 섹션 조사

<Warning title="함정 문제">
JSON 모드만으로는 추출 정확성을 보장할 수 없습니다. JSON Schema를 통한 구조 검증(programmatic enforcement)과 LLM의 의미 검증을 모두 수행해야 합니다. "어느 검증이 더 중요한가?"라는 문제에서 정답은 "둘 다 필수"입니다.
</Warning>
