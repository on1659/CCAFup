## 개요

Tool Design & MCP Integration은 CCA-F 시험의 **18%** 비중을 차지합니다. AI 모델이 외부 도구와 상호작용하는 방법, MCP 프로토콜의 이해, 안전한 도구 설계를 다룹니다.

## 핵심 개념

### 도구 설명이 핵심이다

도구 설명(description)은 **LLM이 도구 선택에 사용하는 주요 메커니즘**입니다.

- 최소한의 설명 → 유사 도구 간 잘못된 선택 발생
- 포함해야 할 내용: 입력 형식, 예시 쿼리, 엣지 케이스, 유사 도구와의 경계
- 모호하거나 겹치는 설명(예: `analyze_content` vs `analyze_document`)은 misrouting 유발

<Tip title="시험 포인트">
"에이전트가 주문 조회 시 get_customer를 호출합니다. 두 도구 모두 설명이 최소합니다. 첫 단계는?" → 정답: **도구 설명을 확장**하여 입력 형식, 예시, 경계를 포함. few-shot 추가나 라우팅 레이어가 아닙니다.
</Tip>

**도구 이름 변경과 분할:**
- 기능 겹침이 있으면 이름을 변경하고 설명을 업데이트 (예: `analyze_content` → `extract_web_results`)
- 범용 도구를 목적별 도구로 분할 (예: `analyze_document` → `extract_data_points`, `summarize_content`, `verify_claim_against_source`)

### MCP 구조화된 에러 응답

MCP의 **isError** 플래그를 사용하여 도구 실패를 에이전트에 전달합니다.

**에러 카테고리:**

| 카테고리 | 설명 | 재시도 가능 |
|---------|------|-----------|
| transient | 타임아웃, 서비스 일시 불가 | Yes |
| validation | 잘못된 입력 | No (입력 수정 필요) |
| business | 정책 위반 | No |
| permission | 권한 부족 | No |

**반환해야 할 에러 메타데이터:**
```json
{
  "isError": true,
  "errorCategory": "transient",
  "isRetryable": true,
  "description": "주문 조회 서비스 타임아웃 (5초 초과)"
}
```

<Warning title="안티패턴">
- 일반적인 "Operation failed" 메시지 → 에이전트가 적절한 복구 결정 불가
- 빈 결과를 성공으로 반환 → 에러 은폐
- 전체 워크플로우를 단일 실패로 종료 → 과도한 반응
</Warning>

### 에이전트 간 도구 분배

에이전트에 **너무 많은 도구**(예: 18개)를 제공하면 선택 신뢰성이 떨어집니다.

| 원칙 | 설명 |
|------|------|
| **역할별 도구 제한** | 각 서브에이전트에 역할에 관련된 도구만 제공 (4-5개 적정) |
| **교차 역할 도구** | 고빈도 필요에만 스코프된 도구 제공 (예: 합성 에이전트에 verify_fact) |
| **제약된 대체** | 범용 도구를 제약된 대안으로 교체 (예: fetch_url → load_document) |

### tool_choice 설정

| 옵션 | 동작 | 사용 시점 |
|------|------|----------|
| `"auto"` | 텍스트 응답 또는 도구 호출 중 모델이 선택 | 기본값 |
| `"any"` | **반드시 도구를 호출** (어떤 도구든 가능) | 구조화된 출력 보장 시 |
| forced `{"type":"tool","name":"..."}` | **특정 도구를 반드시 호출** | 특정 도구가 먼저 실행되어야 할 때 |

<Tip title="시험 핵심">
tool_choice: "any"는 모델이 대화형 텍스트 대신 반드시 도구를 호출하도록 보장합니다. 문서 유형을 모르는 상태에서 여러 추출 스키마 중 선택해야 할 때 사용합니다.
</Tip>

### MCP 서버 스코핑

| 스코프 | 위치 | 공유 |
|--------|------|------|
| 프로젝트 | `.mcp.json` | 팀 공유 (버전 관리) |
| 개인 | `~/.claude.json` | 공유 안 됨 |

- `.mcp.json`에서 환경 변수 확장 지원: `${GITHUB_TOKEN}`
- 모든 설정된 MCP 서버의 도구가 연결 시 동시에 사용 가능
- **MCP Resources**: 콘텐츠 카탈로그(이슈 요약, 문서 계층, DB 스키마)를 노출하여 탐색적 도구 호출 감소

### 내장 도구 선택 기준

| 도구 | 용도 |
|------|------|
| **Grep** | 파일 내용 검색 (함수명, 에러 메시지, import 문) |
| **Glob** | 파일 경로 패턴 매칭 (이름/확장자 패턴) |
| **Read/Write** | 전체 파일 조작 |
| **Edit** | 고유 텍스트 매칭으로 대상 수정 |
| **Bash** | 셸 명령 실행 |

코드베이스 이해 순서: Grep으로 진입점 찾기 → Read로 import 추적 → 흐름 파악 (전체 파일을 한번에 읽지 않음)

## 출제 포인트 요약

1. 도구 설명의 품질이 도구 선택 신뢰성에 미치는 영향
2. MCP isError 플래그와 구조화된 에러 메타데이터 (errorCategory, isRetryable)
3. 에이전트별 도구 분배 원칙과 적정 도구 수 (4-5개)
4. tool_choice 옵션: "auto" vs "any" vs forced selection
5. MCP 서버 스코핑: .mcp.json (프로젝트) vs ~/.claude.json (개인)
6. MCP Resources를 사용한 콘텐츠 카탈로그 노출
7. 내장 도구(Grep, Glob, Read, Write, Edit, Bash) 선택 기준
