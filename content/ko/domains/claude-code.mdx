## 개요

Claude Code 도메인은 CCA-F 시험의 **20%** 비중을 차지합니다. Claude Code CLI의 설정, 최적화, 팀 협업 워크플로우에 대한 실무적 지식을 평가합니다.

## 핵심 개념

### CLAUDE.md 설정 계층

CLAUDE.md는 프로젝트별 규칙과 컨텍스트를 정의하는 핵심 파일입니다.

**설정 계층 (우선순위 낮음 → 높음):**

| 레벨 | 위치 | 공유 방식 | 용도 |
|------|------|----------|------|
| User-level | `~/.claude/CLAUDE.md` | 공유 안 됨 | 개인 설정 |
| Project-level | `.claude/CLAUDE.md` 또는 루트 `CLAUDE.md` | 버전 관리 | 팀 공통 규칙 |
| Directory-level | 하위 디렉토리 `CLAUDE.md` | 버전 관리 | 모듈별 규칙 |

<Tip title="시험 포인트">
새 팀원이 지시를 받지 못하는 문제의 원인 → user-level(~/.claude/CLAUDE.md)에 넣어서 공유가 안 됨. project-level에 넣어야 합니다.
</Tip>

**@import 구문**: 외부 파일을 참조하여 CLAUDE.md를 모듈화합니다. 각 패키지의 CLAUDE.md에서 관련 표준 파일만 선택적으로 포함 가능합니다.

### .claude/rules/ — 경로별 규칙

.claude/rules/ 디렉토리에 YAML frontmatter와 glob 패턴으로 조건부 규칙 파일을 생성합니다.

```yaml
---
paths: ["**/*.test.tsx"]
---
- 테스트 파일은 반드시 describe/it 블록 구조를 사용
- setup/teardown에 beforeEach/afterEach 사용
```

**장점**: 디렉토리 기반 CLAUDE.md보다 유연합니다. 테스트 파일처럼 **코드베이스 전체에 분산된 파일**에 컨벤션을 적용할 때 glob 패턴이 필수입니다.

| 접근 방식 | 적합한 상황 |
|-----------|-----------|
| 서브디렉토리 CLAUDE.md | 특정 디렉토리에만 적용되는 규칙 |
| .claude/rules/ + glob | 파일 유형별 규칙 (여러 디렉토리에 분산) |
| 루트 CLAUDE.md | 프로젝트 전체 공통 규칙 |

### 슬래시 커맨드와 Skills

**슬래시 커맨드 (.claude/commands/):**

| 스코프 | 위치 | 공유 |
|--------|------|------|
| 프로젝트 | `.claude/commands/` | 버전 관리로 팀 공유 |
| 개인 | `~/.claude/commands/` | 공유 안 됨 |

**Skills (.claude/skills/):**

SKILL.md 파일에 frontmatter로 설정합니다:

| 옵션 | 설명 |
|------|------|
| `context: fork` | 격리된 서브에이전트 컨텍스트에서 실행 — 장황한 출력이 메인 대화를 오염시키지 않음 |
| `allowed-tools` | 스킬 실행 중 접근 가능한 도구 제한 (파괴적 작업 방지) |
| `argument-hint` | 인수 없이 호출 시 개발자에게 필요한 파라미터 안내 |

<Warning title="시험 핵심">
Skills(필요 시 호출) vs CLAUDE.md(항상 로드) 선택 기준이 출제됩니다. 작업별 워크플로우는 Skills, 프로젝트 전체 표준은 CLAUDE.md.
</Warning>

### Plan Mode vs Direct Execution

| 상황 | 모드 |
|------|------|
| 대규모 변경, 여러 유효한 접근, 아키텍처 결정 | **Plan mode** |
| 명확한 범위의 단일 파일 버그 수정 | **Direct execution** |
| 코드베이스 탐색 후 구현 | Plan mode → Direct execution 결합 |

- **Explore 서브에이전트**: 장황한 탐색 출력을 격리하고 요약만 메인 컨텍스트에 반환
- `/compact`: 확장된 탐색 세션에서 컨텍스트 사용량 감소

### CI/CD 파이프라인 통합

**비대화형 실행:**
```bash
claude -p "이 PR의 보안 이슈를 분석해줘"
```

| 플래그 | 용도 |
|--------|------|
| `-p` / `--print` | 비대화형 모드 실행, 결과를 stdout 출력 |
| `--output-format json` | 구조화된 JSON 출력 |
| `--json-schema` | 특정 JSON 스키마 강제 |

**세션 컨텍스트 격리**: 코드를 생성한 세션이 자기 변경을 리뷰하면 효과가 떨어집니다. 독립적인 리뷰 인스턴스를 사용하세요.

### 반복적 개선 기법

- **구체적 입출력 예시**: 산문 설명이 일관되지 않을 때 2-3개 예시가 가장 효과적
- **테스트 주도 반복**: 테스트를 먼저 작성 → 테스트 실패를 공유하여 점진적 개선
- **인터뷰 패턴**: Claude에게 질문하게 하여 개발자가 놓친 고려사항을 표면화
- **단일 메시지 vs 순차**: 상호 작용하는 문제는 한 메시지에, 독립적 문제는 순차적으로
