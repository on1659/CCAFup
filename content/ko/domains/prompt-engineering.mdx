## 개요

Prompt Engineering은 CCA-F 시험의 **20%** 비중을 차지합니다. 효과적인 프롬프트 작성, 구조화된 출력 생성, 프롬프트 평가 방법론을 다룹니다.

## 핵심 개념

### 시스템 프롬프트 설계

시스템 프롬프트는 모델의 행동 양식을 정의하는 가장 중요한 요소입니다.

- **역할 정의**: 모델이 수행할 역할을 명확히 지정
- **제약 조건**: 하지 말아야 할 행동을 명시
- **출력 형식**: 응답의 구조와 포맷을 지정
- **컨텍스트**: 필요한 배경 정보 제공

<Callout type="info" title="Best Practice">
시스템 프롬프트는 "무엇을 해야 하는지"보다 "왜 이것을 해야 하는지"를 포함할 때 더 효과적입니다.
</Callout>

### 구조화된 출력

JSON, XML 등 정형화된 포맷으로 출력을 생성하는 기법입니다.

- **JSON 모드**: 유효한 JSON으로 응답을 강제
- **XML 태그**: 구조화된 데이터를 XML로 표현
- **스키마 정의**: 출력 스키마를 사전에 정의하여 일관성 보장

### 프롬프트 기법

| 기법 | 설명 | 장점 |
|------|------|------|
| Zero-shot | 예시 없이 지시만으로 | 간단, 빠른 |
| Few-shot | 2-5개 예시 제공 | 일관된 출력 |
| Chain-of-Thought | 단계별 추론 유도 | 복잡한 문제 해결 |
| Self-consistency | 다중 추론 경로 비교 | 정확도 향상 |

### 프롬프트 테스트 및 평가

프롬프트의 성능을 체계적으로 측정하는 방법입니다.

- **A/B 테스트**: 두 프롬프트의 성능 비교
- **Eval 프레임워크**: 자동화된 프롬프트 평가 파이프라인
- **메트릭**: 정확도, 일관성, 지연 시간, 비용 효율성

### tool_use를 통한 구조화된 출력 (공식 시험 핵심)

tool_use + JSON Schema가 **스키마 준수 구조화된 출력을 보장하는 가장 신뢰할 수 있는 방법**입니다.

- JSON 구문 오류를 구조적으로 제거
- 하지만 **의미적 오류**는 방지하지 못함 (합계가 맞지 않는 항목, 잘못된 필드의 값)

**스키마 설계 핵심:**

| 설계 요소 | 설명 |
|----------|------|
| `required` vs `optional` | 소스에 없을 수 있는 정보는 optional(nullable) — 모델이 환각으로 채우는 것 방지 |
| `enum` + `"other"` | 확장 가능한 카테고리: enum 값 + `"other"` + detail 문자열 |
| `"unclear"` enum 값 | 모호한 경우를 위한 enum 옵션 |

### 검증, 재시도, 피드백 루프

- **Retry-with-error-feedback**: 재시도 시 특정 검증 오류를 프롬프트에 포함하여 모델이 수정하도록 안내
- **재시도 한계**: 정보가 소스 문서에 없으면 재시도가 무효 (형식/구조 오류에만 효과적)
- **detected_pattern 필드**: 개발자가 결과를 기각할 때 패턴을 체계적으로 분석
- **자기 수정 검증**: `calculated_total`과 `stated_total`을 함께 추출하여 불일치 플래그

### Message Batches API

| 특성 | 내용 |
|------|------|
| 비용 절감 | **50%** |
| 처리 시간 | 최대 **24시간** (보장된 지연 SLA 없음) |
| 적합한 사용 | 야간 보고서, 주간 감사, 대량 추출 (지연 허용) |
| 부적합한 사용 | 차단형 pre-merge 검사 (개발자 대기) |
| 멀티턴 도구 호출 | **불가** (단일 요청 내 도구 실행 불가) |
| 결과 상관 | `custom_id` 필드로 요청/응답 매핑 |

<Tip title="시험 핵심">
"두 워크플로우(pre-merge 검사 + 야간 보고서)를 모두 Batches API로 전환하자는 제안을 평가하라" → 정답: 야간 보고서만 배치, pre-merge는 실시간 유지
</Tip>

### 멀티 인스턴스/멀티 패스 리뷰

- **Self-review 한계**: 생성한 세션이 자기 코드를 리뷰하면 생성 시 추론 컨텍스트가 남아 자기 결정을 잘 의문시하지 않음
- **독립 리뷰 인스턴스**: 생성 컨텍스트 없이 리뷰하면 미묘한 이슈를 더 잘 포착
- **멀티 패스 리뷰**: 대규모 리뷰를 파일별 로컬 분석 + 파일 간 통합 분석으로 분할하여 주의 분산 방지

## 출제 포인트 요약

1. tool_use + JSON Schema를 통한 구조화된 출력 보장
2. tool_choice 설정: "auto" vs "any" vs forced selection
3. 스키마의 required vs optional 필드, enum + "other" 패턴
4. Few-shot 예시: 모호한 시나리오, 형식 일관성, false positive 감소
5. 명시적 리뷰 기준 vs 모호한 지시 ("be conservative"는 효과 없음)
6. Message Batches API: 적합 vs 부적합 사용 사례
7. 검증-재시도 루프: 재시도 가능한 오류 vs 재시도 무효 상황
8. 멀티 패스 리뷰로 주의 분산 방지 (파일별 → 통합)
