## 개요

Agentic Architecture는 CCA-F 시험에서 **27%**로 가장 높은 비중을 차지하는 핵심 도메인입니다. AI 에이전트의 설계, 구현, 운영에 관한 전반적인 지식을 다룹니다.

## 핵심 개념

### 에이전트 루프 설계 패턴

에이전트는 기본적으로 **관찰(Observe) → 판단(Think) → 행동(Act)** 루프로 동작합니다. 이 루프의 설계가 에이전트의 성능과 신뢰성을 결정합니다.

- **ReAct 패턴**: Reasoning + Acting을 결합한 가장 기본적인 에이전트 패턴
- **Plan-and-Execute**: 먼저 계획을 세우고 단계별로 실행하는 패턴
- **Reflexion**: 자기 평가를 통해 반복적으로 개선하는 패턴

<Tip title="시험 포인트">
에이전트 루프 설계 시 "언제 루프를 종료할 것인가"에 대한 기준이 자주 출제됩니다. 최대 반복 횟수, 목표 달성 조건, 타임아웃 등 종료 조건을 명확히 이해하세요.
</Tip>

### 오케스트레이션 전략

여러 에이전트 또는 작업을 조율하는 방법입니다.

| 전략 | 설명 | 적합한 상황 |
|------|------|------------|
| Sequential | 순차적으로 하나씩 실행 | 작업 간 의존성이 높을 때 |
| Parallel | 동시에 여러 작업 실행 | 독립적인 작업들 |
| Routing | 조건에 따라 다른 경로로 분기 | 다양한 입력 유형 처리 |

### 멀티 에이전트 시스템

복수의 에이전트가 협력하여 복잡한 작업을 수행하는 아키텍처입니다.

- **Supervisor 패턴**: 하나의 감독 에이전트가 하위 에이전트들을 조율
- **Peer-to-Peer**: 에이전트들이 동등한 위치에서 상호 작용
- **Hierarchical**: 계층적 구조로 역할을 분리

<Warning title="주의">
멀티 에이전트 시스템에서 가장 흔한 실패 원인은 에이전트 간 통신의 모호성입니다. 명확한 인터페이스 정의가 필수입니다.
</Warning>

### 에이전트 감독 및 Guardrails

에이전트의 행동을 제어하고 안전성을 보장하는 메커니즘입니다.

- **입력 Guardrails**: 에이전트에 전달되는 입력을 검증
- **출력 Guardrails**: 에이전트의 출력이 안전한지 확인
- **행동 제한**: 에이전트가 수행할 수 있는 작업의 범위를 제한
- **Human-in-the-Loop**: 중요한 결정에 사람의 승인을 요구

### Claude Agent SDK — 에이전트 루프 구현

공식 시험에서 Agent SDK의 구체적인 구현 패턴이 출제됩니다.

**stop_reason 기반 루프 제어:**

| stop_reason | 의미 | 에이전트 루프 동작 |
|-------------|------|-----------------|
| `"tool_use"` | 모델이 도구 호출을 요청 | 도구 실행 → 결과를 대화에 추가 → **루프 계속** |
| `"end_turn"` | 모델이 최종 응답 완료 | **루프 종료** |

<Warning title="안티패턴">
자연어 신호를 파싱하여 루프 종료를 결정하거나, 임의의 반복 횟수를 주요 종료 메커니즘으로 사용하거나, 어시스턴트 텍스트 콘텐츠를 완료 지표로 확인하는 것은 모두 안티패턴입니다.
</Warning>

**Task 도구로 서브에이전트 생성:**

- `Task` 도구가 서브에이전트를 생성하는 메커니즘
- 코디네이터의 `allowedTools`에 `"Task"`가 반드시 포함되어야 함
- 서브에이전트는 부모 컨텍스트를 자동 상속하지 **않음** — 프롬프트에 명시적으로 제공 필요
- 병렬 실행: 하나의 코디네이터 응답에서 여러 Task 호출을 emit

### Agent SDK Hooks

도구 호출을 가로채어 변환하거나 정책을 강제하는 메커니즘입니다.

- **PostToolUse**: 도구 결과가 모델에 전달되기 전에 변환 (날짜 형식 정규화 등)
- **도구 호출 가로채기**: 정책 위반 행동 차단 (예: $500 초과 환불 차단 → 사람 에스컬레이션)
- hooks는 결정적 보장이 필요할 때 사용, 프롬프트 지시는 확률적 준수만 제공

### 프로그래매틱 강제 vs 프롬프트 기반 가이드

| 구분 | 프로그래매틱 (hooks, 전제 조건) | 프롬프트 기반 |
|------|------|------|
| 신뢰도 | **결정적** — 위반 불가 | 확률적 — 비준수율 존재 |
| 적용 | 핵심 비즈니스 로직, 보안 | 일반적인 워크플로우 순서 |
| 예시 | get_customer 호출 전 process_refund 차단 | "환불 전에 고객 확인을 먼저 하세요" |

<Tip title="시험 핵심">
"에이전트가 12% 확률로 고객 확인을 건너뛰고 환불을 처리합니다. 어떻게 해결할까요?" → 정답: 프로그래매틱 전제 조건 (hooks). 프롬프트 강화가 아닙니다.
</Tip>

### 세션 관리

- `--resume <session-name>`: 특정 세션 이름으로 이전 대화를 이어서 진행
- `fork_session`: 공유 분석 베이스라인에서 독립 분기를 만들어 다른 접근 방식 탐색
- 코드 수정 후 세션을 재개할 때는 변경된 파일에 대해 에이전트에 알려줘야 함
- 오래된 도구 결과가 있는 세션을 재개하기보다 구조화된 요약으로 새 세션을 시작하는 것이 더 신뢰할 수 있음

## 출제 포인트 요약

1. stop_reason 기반 에이전트 루프 제어 (`"tool_use"` vs `"end_turn"`)
2. Task 도구를 사용한 서브에이전트 생성 및 병렬 실행
3. Agent SDK hooks를 통한 도구 호출 가로채기 및 데이터 정규화
4. 프로그래매틱 강제 vs 프롬프트 기반 가이드의 선택 기준
5. 코디네이터-서브에이전트 패턴에서 컨텍스트 전달 방법
6. 태스크 분해: 고정 순차(prompt chaining) vs 동적 적응적 분해
7. 세션 관리: resume, fork_session, 새 세션 시작 판단
