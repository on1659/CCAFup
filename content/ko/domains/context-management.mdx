## 개요

Context Management & Reliability는 CCA-F 시험의 **15%** 비중을 차지합니다. 제한된 컨텍스트 윈도우를 효율적으로 활용하고, AI 시스템의 신뢰성을 보장하는 방법을 다룹니다.

## 핵심 개념

### 컨텍스트 윈도우 최적화

모든 LLM은 처리할 수 있는 토큰 수에 제한이 있습니다. 이를 효율적으로 관리하는 것이 핵심입니다.

- **요약(Summarization)**: 이전 대화를 요약하여 컨텍스트 절약
- **RAG**: 필요한 정보만 검색하여 컨텍스트에 주입
- **슬라이딩 윈도우**: 최근 대화만 유지하고 오래된 것은 제거
- **우선순위 기반**: 중요도에 따라 컨텍스트 할당

<Callout type="info" title="핵심">
컨텍스트 관리의 목표는 "가장 관련 있는 정보를 가장 적은 토큰으로" 전달하는 것입니다.
</Callout>

### 장기 대화 관리

멀티턴 대화에서 맥락을 유지하는 전략입니다.

| 전략 | 설명 | 적합한 상황 |
|------|------|------------|
| Full History | 모든 대화 유지 | 짧은 대화 |
| Sliding Window | 최근 N턴만 유지 | 긴 대화 |
| Summary + Recent | 요약 + 최근 대화 | 복잡한 작업 |
| Memory System | 외부 저장소 활용 | 초장기 대화 |

### 신뢰성 확보 패턴

AI 시스템의 출력을 검증하고 품질을 보장하는 방법입니다.

- **출력 검증(Validation)**: 스키마, 타입, 범위 체크
- **Self-check**: 모델이 자신의 출력을 재검토
- **Cross-validation**: 여러 모델/프롬프트의 결과를 비교
- **Confidence Score**: 모델의 확신도를 기반으로 판단

<Warning title="주의">
"모델이 정확하다고 확신할수록" 실제로는 할루시네이션일 가능성을 배제하지 마세요. 외부 검증 없는 Self-check만으로는 신뢰성을 보장할 수 없습니다.
</Warning>

### 에러 처리 및 폴백

- **Graceful Fallback**: 주 경로 실패 시 대안 경로 제공
- **사용자 알림**: 실패를 투명하게 알리고 대안 제시
- **Retry with Modification**: 프롬프트를 수정하여 재시도

### Lost-in-the-Middle 효과 (공식 시험 출제)

모델은 긴 입력의 **시작과 끝** 부분의 정보는 잘 처리하지만, **중간 섹션의 결과를 놓칠 수 있습니다**.

**완화 전략:**
- 핵심 발견 요약을 **입력 시작에 배치**
- 상세 결과를 명시적 **섹션 헤더**로 조직
- 핵심 데이터(금액, 날짜, 주문번호)를 별도 "case facts" 블록으로 추출

### 도구 결과의 컨텍스트 관리

도구 결과가 컨텍스트에 축적되면 관련성에 비례하지 않는 토큰을 소비합니다.

- 예: 주문 조회 시 40개+ 필드 반환 → 실제 필요한 건 5개
- **해결**: 장황한 도구 출력을 관련 필드만으로 트리밍한 후 컨텍스트에 포함

### 에스컬레이션 및 모호성 해결 (공식 시험 핵심)

**적절한 에스컬레이션 트리거:**
- 고객이 명시적으로 사람을 요청할 때 → **즉시** 에스컬레이션 (먼저 해결 시도 금지)
- 정책 예외/공백 → 에스컬레이션
- 의미 있는 진전을 이룰 수 없을 때 → 에스컬레이션

**부적절한 에스컬레이션 기준:**
- 감정 기반 에스컬레이션 → 감정 ≠ 사례 복잡도
- LLM 자가 보고 자신감 점수 → 잘 보정되지 않음

**다중 고객 매칭 처리:**
- 여러 고객이 매칭되면 → 추가 식별자를 요청 (휴리스틱 선택 금지)

### 멀티에이전트 에러 전파

| 안티패턴 | 올바른 접근 |
|---------|-----------|
| 에러를 조용히 억제 (빈 결과를 성공으로) | 구조화된 에러 컨텍스트 반환 |
| 단일 실패로 전체 워크플로우 종료 | 부분 결과로 계속 + 커버리지 갭 표시 |
| 일반적 "search unavailable" 상태 | 실패 유형, 시도한 쿼리, 부분 결과, 대안 포함 |

### Scratchpad와 컨텍스트 경계 관리

- **Scratchpad 파일**: 긴 세션에서 핵심 발견을 파일에 기록, 후속 질문에서 참조
- **서브에이전트 위임**: 장황한 탐색을 서브에이전트에 위임, 메인 에이전트는 하이레벨 조율 유지
- **구조화된 상태 내보내기**: 크래시 복구를 위해 각 에이전트가 상태를 알려진 위치에 저장

### 정보 출처 보존 (Provenance)

- 요약 과정에서 소스 귀속(source attribution)이 손실됨
- **claim-source 매핑**: 서브에이전트가 소스 URL, 문서명, 관련 발췌, 출판일을 구조화 출력에 포함
- **상충하는 정보**: 하나를 선택하지 말고 양쪽 소스를 귀속과 함께 표시
- **시간 데이터**: 출판/수집 날짜를 포함하여 시간적 차이가 모순으로 오해되는 것 방지

## 출제 포인트 요약

1. Lost-in-the-middle 효과와 완화 (시작에 핵심 배치, 섹션 헤더)
2. 도구 출력 트리밍 — 관련 필드만 컨텍스트에 유지
3. 에스컬레이션 기준: 명시적 요청 즉시 에스컬레이션, 감정/자신감 점수는 부적절
4. 구조화된 에러 전파: 실패 유형, 부분 결과, 대안 포함
5. Scratchpad와 서브에이전트를 통한 긴 세션 컨텍스트 관리
6. 정보 출처 보존: claim-source 매핑, 상충 정보 처리, 시간 데이터
7. 인간 리뷰 워크플로우: 필드별 신뢰도 점수, 층화 표본 추출, 문서 유형별 정확도 검증
