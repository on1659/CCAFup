## 문제 구조 — 일관된 3단계 형식

CCA-F의 모든 문제는 동일한 구조를 따릅니다:

```
[시나리오 앵커] → [구체적 증거/데이터] → [가장 효과적인 해결책은?]
```

### 1단계: 시나리오 앵커링

6개 시나리오 중 4개가 랜덤 선택되어 시험에 출제됩니다. 모든 문제는 반드시 하나의 시나리오에 고정되어 있습니다.

> "당신은 Claude Agent SDK를 사용하여 고객 지원 에이전트를 구축하고 있습니다. 에이전트는 반품, 청구 분쟁, 계정 문제 같은 높은 모호성의 요청을 처리합니다..."

### 2단계: 구체적 증거 제시

문제는 반드시 **구체적인 수치, 로그, 관찰 결과**를 포함합니다. 추상적인 이론 질문은 없습니다.

| 증거 유형 | 실제 예시 |
|----------|---------|
| 프로덕션 데이터 | "12%의 케이스에서 에이전트가 get_customer를 건너뜁니다" |
| 성과 지표 | "55% 1차 해결률로 80% 목표에 미달합니다" |
| 시스템 로그 | "코디네이터가 주제를 시각 예술 3개로만 분해했습니다" |
| 관찰 결과 | "14개 파일 리뷰에서 일관성 없는 피드백이 나옵니다" |

### 3단계: "가장 효과적인" 해결책

질문은 항상 **"most effectively"**, **"most effective first step"**, **"best enables"** 형태입니다. 올바른 것이 아니라 **가장 적절한** 것을 묻습니다.

## 선택지 패턴 — 4개 중 3개가 안티패턴

공식 12문항을 분석하면 오답의 유형이 일정합니다:

| 오답 유형 | 설명 | 출현 빈도 |
|----------|------|----------|
| **프롬프트 함정** | "시스템 프롬프트에 추가한다", "few-shot 예시를 넣는다" | 거의 매 문제 |
| **과도한 엔지니어링** | 별도 분류 모델, ML 인프라, 라우팅 레이어 | 자주 |
| **다른 문제 해결** | 감정 분석, 선제적 캐싱 등 근본 원인과 무관한 접근 | 자주 |
| **극단적 반응** | 전체 시스템 중단, 무한 재시도, 에러 무시 | 가끔 |

<Tip title="핵심 전략">
선택지를 볼 때 "이것이 근본 원인을 직접 해결하는가?"를 물으세요. 근본 원인에 비례적으로 대응하는 선택지가 정답입니다. 과도하거나 부족한 대응은 오답입니다.
</Tip>

## 5대 출제 공식

시험 전체에서 반복되는 5가지 패턴입니다. 이것만 알면 대부분의 함정을 피할 수 있습니다.

### 공식 1: 보장이 필요한 행동 → 프로그래매틱 강제

**문제 패턴**: "에이전트가 X%의 확률로 필수 단계를 건너뜁니다"

| 함정 | 정답 |
|------|------|
| 시스템 프롬프트 강화 | **hooks 또는 프로그래매틱 전제 조건** |
| few-shot 예시 추가 | **코드로 차단 (블로킹 게이트)** |

> 비즈니스 규칙(환불 전 고객 확인, $500 초과 승인)에는 확률적 준수가 아닌 결정적 보장이 필요합니다.

### 공식 2: 도구 선택 오류 → 도구 설명 개선

**문제 패턴**: "에이전트가 잘못된 도구를 선택합니다. 두 도구의 설명이 최소합니다"

| 함정 | 정답 |
|------|------|
| few-shot으로 올바른 도구 선택 예시 | **도구 설명 확장** (입력 형식, 예시, 경계) |
| 라우팅 분류기 추가 | **이름 변경 + 설명 차별화** |

> 도구 설명이 LLM의 도구 선택 주요 메커니즘입니다. 설명이 부실하면 아무리 좋은 프롬프트도 소용없습니다.

### 공식 3: 대규모 리뷰 품질 → 멀티패스 분할

**문제 패턴**: "N개 파일을 한번에 리뷰하면 일관성 없는 결과"

| 함정 | 정답 |
|------|------|
| 더 큰 컨텍스트 윈도우 모델 | **파일별 로컬 분석 + 파일 간 통합 패스** |
| 3회 독립 리뷰 후 투표 | **포커스된 패스로 분할** |

> 더 큰 컨텍스트 윈도우는 주의 품질(attention quality) 문제를 해결하지 못합니다.

### 공식 4: CI/CD에서 Claude Code 대기 → -p 플래그

**문제 패턴**: "파이프라인이 무기한 대기합니다"

| 함정 | 정답 |
|------|------|
| CLAUDE_HEADLESS=true 환경변수 | **`-p` (--print) 플래그** |
| --batch 플래그 | **비대화형 모드** |
| stdin 리다이렉트 | 존재하지 않는 기능들은 함정 |

### 공식 5: 비용 절감 → 차단형은 실시간, 비차단만 배치

**문제 패턴**: "두 워크플로우를 모두 Batches API로 전환하자는 제안"

| 함정 | 정답 |
|------|------|
| 둘 다 배치 전환 | **야간 보고서만 배치, pre-merge는 실시간** |
| 둘 다 실시간 유지 | Batches API는 24시간 SLA, 차단형에 부적합 |

## 공식 해설이 알려주는 오답 패턴

공식 12문항의 오답 해설에서 반복되는 핵심 문구:

| 공식 해설 문구 | 의미 |
|-------------|------|
| "relies on **probabilistic LLM compliance**" | 프롬프트 기반 접근의 한계 — 보장 불가 |
| "**over-engineered**, requiring labeled data and ML infrastructure" | 프롬프트 최적화를 먼저 시도하지 않음 |
| "solves **a different problem entirely**" | 근본 원인이 아닌 다른 것을 해결 |
| "**larger context windows don't solve attention quality** issues" | 양이 아닌 질의 문제 |
| "**generic status hides valuable context**" | 구조화된 에러의 중요성 |
