### Prisma + Supabase 역할 분리 ← Round 2 추가

Prisma는 `service_role` key로 연결하면 RLS를 우회한다. 따라서 역할을 분리한다:

| 용도 | 도구 | 키 | RLS |
|------|------|-----|-----|
| Cron, 관리자 API, 마이그레이션 | **Prisma** | service_role | 우회 (의도적) |
| 유저 대면 쿼리 (점수, 북마크, 글쓰기) | **Supabase Client SDK** | anon key + JWT | 적용 |

이렇게 하면 Prisma의 타입 안전성과 Supabase의 RLS 보안을 동시에 활용할 수 있다.

### 스키마 관리 역할 분리 ← Round 6 추가

| 관리 대상 | 도구 | 파일 |
|----------|------|------|
| 테이블, 컬럼, 인덱스 | **Prisma Migrate** | `prisma/schema.prisma` |
| RLS 정책 | **별도 SQL 파일** | `supabase/migrations/rls_*.sql` |
| Seed 데이터 (초기 문제 풀 등) | **Prisma Seed** | `prisma/seed.ts` |

Prisma Migrate 후 RLS SQL을 수동 적용하는 순서. CI에서 자동화 가능.

### 렌더링 & 캐싱 전략 ← Round 5 추가

### 성능 예산 (Performance Budget) ← Round 13 추가

| 지표 | 목표 | 측정 |
|------|------|------|
| LCP (Largest Contentful Paint) | < 2.5s | SSG/ISR 페이지 기준 |
| FID (First Input Delay) | < 100ms | 퀴즈 인터랙션 기준 |
| CLS (Cumulative Layout Shift) | < 0.1 | 폰트 로딩, 이미지 placeholder |
| JS 번들 (초기 로드) | < 150KB (gzipped) | 정적 페이지 기준 |
| JS 번들 (퀴즈 페이지) | < 250KB (gzipped) | Fuse.js, 타이머 등 포함 |
| TTI (Time to Interactive) | < 3.5s | 모바일 4G 기준 |

**번들 최적화 전략:**
- Shiki: Server Component에서 렌더링 (클라이언트 번들에 포함 안 됨)
- satori: Route Handler에서만 사용 (클라이언트 번들에 포함 안 됨)
- Fuse.js: 검색 페이지에서만 dynamic import
- cmdk: Cmd+K 트리거 시 lazy load

| 페이지 유형 | 렌더링 | revalidate | 이유 |
|------------|--------|------------|------|
| 도메인 상세, 시나리오, FAQ, 치트시트 | **SSG** | 빌드 시 생성 | MDX 정적 콘텐츠, 변경 시 재빌드 |
| 홈 (Daily Challenge, Signal) | **ISR** | 3600초 (1시간) | Cron이 매일 업데이트하므로 1시간 캐싱 충분 |
| 모의시험 퀴즈 | **SSR** | - | 유저별 랜덤 문제 추출, 캐싱 불가 |
| 대시보드 | **SSR** | - | 유저별 데이터, 캐싱 불가 |
| 커뮤니티 목록 | **ISR** | 60초 | 새 글 반영 빈도와 서버 부하 균형 |
| Signal 피드 | **ISR** | 1800초 (30분) | Cron이 1일 1회이므로 여유 |
| 용어 사전 | **SSG** | 빌드 시 생성 | JSON/MDX 정적 데이터 |

| 콘텐츠 유형 | 저장 방식 | 이유 |

## 자동화 파이프라인

| 작업 | 실행 | 도구 | 모델 |
|------|------|------|------|
| Daily Signal 수집 | 매일 06:00 KST | Railway Cron + Node.js script | Claude Haiku (요약) |
| 오늘의 문제 선택 | 매일 06:00 KST | Railway Cron | 문제 풀에서 가중 랜덤 |
| AI 문제 생성 | 주 1회 | Railway Cron | Claude Sonnet (출제) → 관리자 검수 |
| Weekly Digest | 매주 월요일 09:00 KST | Railway Cron | Claude Haiku (요약) |
| 시험 가이드 PDF diff | 주 1회 | Railway Cron + diff script | 텍스트 비교 (LLM 불필요) |
| 이메일 뉴스레터 발송 | Daily/Weekly | **Resend** (이메일 API) | - |

> **이메일 구독 해지 흐름** ← Round 15 추가
> - 모든 이메일 하단에 unsubscribe 링크 포함 (법적 필수)
> - 링크: `{SITE_URL}/api/unsubscribe?token={signed_token}`
> - Route Handler에서 token 검증 → subscribers 테이블의 unsubscribed_at 업데이트
> - 해지 확인 페이지 렌더링

### 테스트 전략 ← Round 11 추가

| 유형 | 도구 | 대상 |
|------|------|------|
| 단위 테스트 | **Vitest** | 퀴즈 점수 계산, 스트릭 로직, 도메인 가중 랜덤, 유틸 함수 |
| 통합 테스트 | **Vitest + @supabase/supabase-js** | Server Actions, API 엔드포인트, Cron 로직 |
| E2E 테스트 | **Playwright** | 프록터링 모드 (전체화면, 타이머), 인증 플로우, 퀴즈 응시 플로우 |
| 타입 체크 | **tsc --noEmit** | CI에서 빌드 전 타입 검증 |

MVP에서는 단위 테스트(Vitest) + 타입 체크만 필수. E2E는 프록터링 모드 구현 시 추가.

### 로컬 개발 환경 ← Round 9 추가

```bash
# 1. 프로젝트 클론 & 의존성 설치
git clone https://github.com/on1659/ccafup.git
cd ccaf-hub
pnpm install

# 2. 환경변수 설정
cp .env.example .env.local
# .env.local에 Supabase URL, anon key, service_role key 입력

# 3. DB 마이그레이션 (Supabase Cloud 직접 사용)
pnpm prisma migrate dev

# 4. Seed 데이터 (초기 문제 풀)
pnpm prisma db seed

# 5. 개발 서버 실행
pnpm dev
```

로컬 DB는 Supabase Cloud 개발 프로젝트를 직접 사용한다 (Supabase CLI 로컬 인스턴스는 Auth 테스트가 번거로우므로). Supabase에서 개발용 프로젝트를 별도 생성하여 프로덕션과 분리.

### .env.example 템플릿 ← Round 9 추가

```env
# Supabase (Public)
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=

# Supabase (Server only)
SUPABASE_SERVICE_ROLE_KEY=
DATABASE_URL=

# Anthropic (Server only - Cron)
ANTHROPIC_API_KEY=

# Resend (Server only - Email)
RESEND_API_KEY=

# Upstash Redis (Server only - Rate Limit)
UPSTASH_REDIS_REST_URL=
UPSTASH_REDIS_REST_TOKEN=

# PostHog (Public)
NEXT_PUBLIC_POSTHOG_KEY=
NEXT_PUBLIC_POSTHOG_HOST=

# Sentry
SENTRY_DSN=
```

### CLAUDE.md 프로젝트 룰 ← Round 9 추가

프로젝트 루트에 `CLAUDE.md`를 배치하여 Claude Code가 코딩 컨벤션을 자동 적용하도록 한다.

핵심 룰 (상세는 프로젝트 생성 시 작성):
- 파일명: kebab-case.tsx, export: PascalCase
- 색상 하드코딩 금지 — CSS 변수 또는 Tailwind 클래스만 사용
- DB 접근: 유저 대면 = Supabase Client SDK, 관리자/Cron = Prisma
- Server Actions에서 인증 체크 필수 (createClient → getUser)
- MDX 콘텐츠는 content/{locale}/ 하위에 배치
- 커밋 메시지: conventional commits (feat/fix/chore)

## 보안 ← Round 4 추가

### Rate Limiting

| 항목 | 방식 |
|------|------|
| 기여 폼 (나도 출제하기, 용어 제안) | **upstash/ratelimit** — IP당 시간당 3회 |
| 커뮤니티 글쓰기 | IP당 분당 5회 |
| 퀴즈 API | 인증 유저당 분당 30회 |
| 비인증 API | IP당 분당 10회 |

### CSRF 보호

| 경로 | 방식 |
|------|------|
| Server Actions (Next.js) | 내장 CSRF 보호 (자동) |
| Supabase Client SDK 직접 호출 | `@supabase/ssr`의 세션 쿠키 기반 인증으로 보호 |

### 환경변수 분류

```
# ✅ 클라이언트 노출 가능 (NEXT_PUBLIC_ 접두사)
NEXT_PUBLIC_SUPABASE_URL=https://xxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ...

# ❌ 서버 전용 (절대 클라이언트 노출 금지)
SUPABASE_SERVICE_ROLE_KEY=eyJ...   # Prisma, Cron에서만 사용
DATABASE_URL=postgresql://...       # Prisma에서만 사용
ANTHROPIC_API_KEY=sk-...           # Cron 파이프라인에서만 사용
RESEND_API_KEY=re_...              # 이메일 발송에서만 사용
```

### Supabase 무료 티어 제한 ← Round 3 추가

| 리소스 | 제한 | 대응 |
|--------|------|------|
| DB | 500MB | Signal 데이터 90일 보존 후 아카이브 (summary만 유지, 원문 삭제) |
| Storage | 1GB | 커뮤니티 업로드 이미지 2MB 제한 + WebP 변환 |
| Bandwidth | 2GB/월 | ISR/SSG로 정적 페이지 캐싱, API 호출 최소화 |
| Auth MAU | 50,000 | MVP에 충분, 초과 시 Pro 전환 |
| Edge Functions | 500K 호출 | 사용 안 함 (Server Actions 사용) |

### 벤더 종속성 & 마이그레이션 경로 ← Round 12 추가

| 서비스 | 종속 수준 | 대안 경로 |
|--------|----------|----------|
| Supabase Auth | 높음 | NextAuth.js로 전환 가능 (OAuth 설정만 이동) |
| Supabase PostgreSQL | 낮음 | 표준 PostgreSQL — Railway PG, Neon 등 어디든 이전 가능 |
| Supabase Storage | 중간 | Cloudflare R2 또는 AWS S3로 URL 패턴만 변경 |
| Upstash Redis | 낮음 | Redis 프로토콜 표준 — 자체 Redis 또는 다른 호스팅 |
| Railway | 낮음 | Docker 기반 — Fly.io, Render 등 어디든 이전 가능 |

**원칙:** Supabase 전용 기능(Edge Functions, Realtime, Supabase Vector) 사용 금지. PostgreSQL 표준 + 표준 OAuth만 사용하여 탈출 비용 최소화.

**Next.js 업그레이드 원칙:** `next.config.js`에서 experimental 플래그 사용 최소화. App Router 코어 API(Server Components, Server Actions, Route Handlers)만 사용.
